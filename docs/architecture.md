# Архітектура MLOps платформи

## Огляд системи

MLOps-платформа — це інтегроване середовище для керування життєвим циклом моделей машинного навчання: від експериментів і тренування до реєстрації артефактів, розгортання та спостереження за інференсом у production.

**Призначення:** забезпечити передбачувані, відтворювані та безпечні поставки ML-робочих навантажень на Kubernetes у хмарі Google Cloud Platform (GCP), з єдиною точкою правди в Git і централізованою обсервабіліті.

**Проблеми, які вирішує:**

- Розрив між дослідженням і production (однакові підходи до версіонування коду, даних і моделей).
- Ручні або непрозорі деплої — заміна на декларативний GitOps-потік.
- Відсутність видимості затримок, помилок і вартості — вбудований моніторинг і квоти ресурсів.
- Складність керування секретами та доступом до хмарних сервісів — Workload Identity та External Secrets Operator.

## Принципи проектування

| Принцип | Зміст |
|--------|--------|
| **GitOps-first** | Бажаний стан кластера описується в Git; Flux CD синхронізує його з Kubernetes. Зміни проходять через review, історія змін збережена. |
| **Infrastructure as Code (IaC)** | Мережа, кластер, IAM, реєстри артефактів і сховища задаються через Terraform — відтворювані середовища без «ручних» кліків у консолі. |
| **Observability-first** | Метрики, логи та алерти (kube-prometheus-stack, Grafana) закладені з початку, а не додаються постфактум. |
| **Cost awareness** | Свідомий вибір типів нод (у т.ч. spot), масштабування до нуля де доречно, квоти та ліміти ресурсів зменшують непередбачувані витрати. |

## Компоненти

### Terraform (GCP)

Опис інфраструктури: **VPC**, **GKE** (приватний кластер), **IAM** (ролі та сервісні акаунти), **Artifact Registry** (контейнери та пакети), **GCS** (дані, артефакти, бекапи залежно від політики проєкту).

### Flux CD (GitOps delivery)

Безперервна синхронізація маніфестів і Helm-релізів із Git-репозиторію в кластер; політики оновлень і можливість тимчасового призупинення через suspend.

### Helm charts

- **llm-serving** — розгортання сервісу інференсу LLM (наприклад, поверх vLLM).
- **ml-gateway** — шлюз до ML/LLM API (маршрутизація, обмеження, інтеграція з моніторингом).
- **grafana-dashboards** — дашборди для команди (латентність, GPU, черги тощо).

### Airflow 2

Оркестрація **ML pipelines**: ETL, тренування, пост-обробка, виклики зовнішніх сервісів; DAG-и як код, планування і залежності між задачами.

### MLflow

**Experiment tracking** (параметри, метрики, артефакти запусків) та **model registry** (стадії моделі, промоушн версій до production).

### vLLM

Високопродуктивний **LLM inference** на GPU з ефективним батчингом і кешуванням KV; типовий бекенд для llm-serving.

### kube-prometheus-stack

Повний стек **моніторингу**: Prometheus, Alertmanager, Grafana, стандартні експортери для Kubernetes; основа для SLO/алертів.

### External Secrets Operator

Синхронізація секретів з зовнішніх сховищ (наприклад, Google Secret Manager) у Kubernetes `Secret` без збереження відкритих значень у Git.

### NVIDIA GPU Operator

Керування драйверами, runtime та device plugin для **GPU** у GKE; спрощує узгоджені оновлення стеку GPU на нодах.

## Потік даних: від тренування до production inference

1. **Дані** надходять у pipeline (Airflow) або зберігаються в GCS / інших сховищах згідно з архітектурою даних.
2. **Експерименти** фіксуються в MLflow: код, гіперпараметри, метрики, артефакти.
3. Успішна версія **реєструється** в MLflow Model Registry і отримує стадію (наприклад, Staging → Production) за погодженим процесом.
4. **Контейнер** з моделлю або сервісом збирається й публікується в Artifact Registry; маніфести оновлюються в Git.
5. **Flux** застосовує зміни в кластері; **llm-serving** / **ml-gateway** обслуговують трафік через **vLLM** (для LLM) або відповідні deployment-и.
6. **Метрики та логи** збираються kube-prometheus-stack і дашбордами Grafana; алерти сповічають про деградації.

## Мережева архітектура

- **Private GKE:** вузли та/або control plane без публічних endpoint-ів де це налаштовано політикою; доступ через приватні підмережі та авторизовані канали.
- **Cloud NAT:** вихідний трафік з приватних інстансів до інтернету без публічних IP на подах; передбачуваний контроль egress.
- **Namespace isolation:** логічне розділення команд або середовищ (dev/stage/prod) через namespaces, мережеві політики та квоти; зменшення радіуса впливу збоїв і помилкових деплоїв.

## Безпека

- **Workload Identity:** поди отримують ідентичність GCP без довгоживучих JSON-ключів; доступ до GCS, Secret Manager тощо через IAM.
- **NetworkPolicy:** обмеження трафіку між подами/неймспейсами; принцип мінімальних привілей на рівні мережі.
- **External Secrets:** секрети не комітяться в Git; джерело істини — кероване сховище, Kubernetes лише отримує синхронізовані об’єкти.

## Оптимізація вартості

- **Spot instances (преривальні ВМ):** зниження ціни для толерантних до витіснення навантажень (тренування, батчі); критичний інференс може залишатися на стабільних пулах.
- **Scale-to-zero:** автоматичне зменшення реплік до нуля для сервісів без навантаження (з урахуванням cold start і SLA).
- **Resource quotas та limits:** верхні межі CPU/GPU/пам’яті на namespace або проєкт; запобігання «забігу» витрат і перегріву кластера.
