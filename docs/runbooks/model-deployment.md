# Ранбук: Деплой нової моделі

Покроковий процес виведення **нової версії моделі** в production на MLOps-платформі (MLflow, Artifact Registry, GitOps, Kubernetes).

## Симптоми / тригери

- Завершено тренування або fine-tuning; нова версія пройшла валідацію в **MLflow**.
- Запит бізнесу на оновлення моделі в production.
- Плановий реліз (наприклад, щомісячне оновлення ваг або образу).

## Діагностика (перед деплоєм)

Переконатися, що нова версія готова і відтворювана.

### MLflow

- Перевірити **run**: метрики, артефакти, commit hash коду (якщо логується).
- У **Model Registry** перевірити стадію, підпис моделі (model signature), залежності.

### Артефакт і образ

- Версія образу в **Artifact Registry** відповідає тегу з CI.
- Якщо модель монтується з **GCS** — шлях і контрольна сума / версія об’єкта задокументовані.

### Kubernetes (поточний стан)

```bash
kubectl get deployment -n NAMESPACE -l app=SERVICE_NAME -o wide
kubectl rollout history deployment/DEPLOYMENT_NAME -n NAMESPACE
```

### Моніторинг (базова лінія)

У **Grafana** зафіксувати поточні метрики латентності, помилок і навантаження для порівняння після релізу.

## Рішення

### 1. Реєстрація та промоушн у MLflow

1. Зареєструвати модель або нову версію в **MLflow Model Registry**.
2. Перевести версію в **Staging**; виконати smoke-тести на staging-кластері або namespace.
3. Після підтвердження — промоушн до **Production** (або еквівалентна стадія за вашою політикою).

### 2. Оновлення контейнера та маніфестів

1. Зібрати образ з новою моделлю (або entrypoint, що тягне артефакт за версією).
2. Запушити тег у **Artifact Registry**.
3. Оновити у **Git** значення образу (Helm `values`, Kustomize patch або маніфест deployment) — версія тега має бути явною, не `latest` для production.

### 3. GitOps: застосування змін

1. Створити MR/PR; code review.
2. Після мерджу **Flux** (або ваш пайплайн) синхронізує стан кластера.
3. Перевірити **Kustomization** / **HelmRelease** на помилки:

```bash
flux get kustomizations -A
flux get helmreleases -A
```

### 4. Перевірка rollout

```bash
kubectl rollout status deployment/DEPLOYMENT_NAME -n NAMESPACE
kubectl get pods -n NAMESPACE -l app=SERVICE_NAME
```

### 5. Canary / blue-green (якщо налаштовано)

1. Спочатку направити частку трафіку на нову версію через **ml-gateway** або другий deployment.
2. Порівняти метрики; при успіху — повний перехід; при регресії — відкат.

### 6. Відкат

```bash
kubectl rollout undo deployment/DEPLOYMENT_NAME -n NAMESPACE
```

Або відкотити коміт у Git і дати **Flux** синхронізувати попередній тег образу.

### 7. Постреліз

- Оновити внутрішню документацію: версія моделі, MLflow run ID, тег образу.
- Переконатися, що алерти в **Alertmanager** не спрацьовують хибно.

## Превенція

- Єдиний шлях деплою через **Git** + **Flux**; заборона ручного `kubectl set image` у production без наступного відображення в Git.
- Обов’язковий **MR** з чеклістом: метрики, тести, відкат.
- Версіонування моделей у **MLflow** узгоджене з тегами образів або параметрами завантаження артефакту.
- Автоматичні перевірки в CI: лінт маніфестів, наявність ресурсних лімітів, скан образу.
- Регулярні навчальні імітації відкату (**rollout undo** + перевірка метрик).
